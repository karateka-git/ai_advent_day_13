# Development Policy

## Совместимость со старыми версиями

В этом проекте по умолчанию не нужно добавлять или сохранять обратную совместимость со старыми версиями, форматами состояния, старыми JSON-структурами или legacy-поведением, если пользователь явно не попросил об этом.

Правило для Codex:
- если в ходе реализации возникает идея сохранить совместимость со старой версией, старым storage-форматом или legacy API, сначала нужно отдельно спросить пользователя, нужна ли эта совместимость;
- без явного подтверждения пользователя нужно предпочитать более чистую текущую реализацию без compatibility-слоя.

## Напоминание Про Коммит

Если в проекте есть незакоммиченные изменения и пользователь, похоже, переключается на другую задачу, новый этап работы или другой смысловой блок, нужно мягко предложить сделать коммит перед продолжением.

Правило для Codex:
- не навязывать коммит после каждой мелкой правки;
- предлагать коммит именно в момент естественного переключения контекста;
- формулировать это как рекомендацию, а не как обязательное требование;
- если пользователь явно не хочет коммитить, просто продолжать работу без спора.

Локальное усиление правила для этого проекта:
- если есть незакоммиченные изменения и пользователь просит новый смысловой блок, который требует новых изменений в коде или файлах, нужно явно остановиться и напомнить про коммит;
- если пользователь просит только анализ, объяснение, проверку, запуск команд, сборку, просмотр логов или ручной сценарий без новых правок в коде/файлах, отдельно останавливать работу из-за коммита не нужно;
- до ответа пользователя на это напоминание не переходить к следующей реализации с новыми изменениями в коде или файлах;
- продолжать работу можно только после явного подтверждения пользователя, что он либо хочет сделать коммит сейчас, либо осознанно хочет продолжать без коммита.

## Smoke-Check Проекта

Для ручной проверки основной пользовательской функциональности проекта нужно использовать проектный сценарий:
- [manual-smoke-checklist.md](/C:/Users/compadre/Downloads/Projects/AiAdvent/day_12/docs/manual-smoke-checklist.md)

Правило для Codex:
- если пользователь просит `проверить проект`, `прогнать основной сценарий`, `сделать smoke-check` или формулирует близкую просьбу без дополнительного уточнения, по умолчанию ориентироваться на этот чеклист;
- comparison runner не включать в такой smoke-check, если пользователь явно не попросил проверить dev-сценарии.

Практические нюансы после первого прогона smoke-check:
- versioned scripted-сценарии должны лежать в репозитории, а не в `build/`; для них использовать каталог `scripts/smoke-check/scenarios/`;
- `build/smoke-check/` использовать только для артефактов прогонов: вывода, JSON-снимков и временных результатов;
- для изолированной проверки каждой стратегии сначала очищать `config/conversations/*`, иначе сценарии будут использовать уже существующие persisted state;
- перед новым scripted smoke-check очищать старые артефакты в `build/smoke-check/`, если они больше не нужны, чтобы результаты прошлых прогонов не смешивались с текущими;
- допустимо удалять старые выходные файлы и JSON-снимки из `build/smoke-check/` перед новым прогоном, если пользователь не просил их сохранить отдельно;
- для scripted-прогона удобно повторно вызывать `use timeweb`, чтобы заново выбрать стратегию и не зависеть от другого провайдера;
- при захвате вывода `.bat` через scripted stdin в текущей консоли кириллица может отображаться кракозябрами; в таком случае ориентироваться не на текстовый захват, а на фактическое поведение команд и проверку JSON;
- для redirected stdin приложение теперь должно читать scripted-ввод как UTF-8, а helper `scripts/run-scripted-session.ps1` должен писать сценарий в `stdin` без BOM;
- если scripted smoke-check ведёт себя так, как будто первая команда стала обычным пользовательским сообщением, сначала проверить, что запущен свежий `installDist`, а не старая собранная версия;
- после изменений в `Main.kt`, `scripts/run-scripted-session.ps1` или любой логике чтения/обработки консольного ввода scripted smoke-check нужно гонять только после повторного:
  - `.\gradlew.bat build`
  - `.\gradlew.bat installDist`
- старый `build/install/ai_advent_day_12/bin/ai_advent_day_12.bat` после таких правок считать недостоверным для проверки сценариев;
- scripted smoke-check нельзя запускать параллельно: сценарии используют общий каталог `config/conversations`, и параллельный запуск приводит к блокировкам файлов и смешиванию результатов;
- при проверке persisted state помнить ожидаемое поведение: `no_compression` и `sliding_window` не обязаны сохранять `strategyState`, а `summary`, `sticky_facts` и `branching` должны сохранять свой типизированный `strategyState`;
- layered JSON в проекте считается разреженным:
  - `shortTerm` ожидается всегда;
  - `working`, `longTerm` и `pending` могут отсутствовать в persisted JSON, если слой пустой;
  - отсутствие пустого слоя не считать ошибкой smoke-check;
- после внедрения pending-confirmation flow при проверке layered memory учитывать, что часть фактов может попасть не сразу в `working` или `longTerm`, а в `pending`.
