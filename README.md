# day_12 — Явная модель памяти ассистента

`day_12` вырос из [day_10](C:/Users/compadre/Downloads/Projects/AiAdvent/day_10), где уже был CLI-агент с набором short-term стратегий памяти, сравнением стратегий и сохранением истории диалога.

Главная задача этого этапа была другой: не просто хранить историю, а ввести **явную модель памяти ассистента**, где разные типы знаний разделены по смыслу и управляются отдельно.

Исходное задание:

> День 11. Модель памяти ассистента
>
> Опишите и реализуйте модель памяти для ассистента.
>
> Разделите информацию минимум на 3 типа:
>
> - краткосрочная (текущий диалог)
> - рабочая (данные текущей задачи)
> - долговременная (профиль, решения, знания)
>
> Сделайте так, чтобы:
>
> - разные типы памяти хранились отдельно
> - вы явно выбирали, что и куда сохраняется
>
> Проверьте:
>
> - какие данные попадают в каждый слой
> - как это влияет на ответы ассистента
>
> Результат:
>
> Агент с явной моделью памяти (memory layers)

## Что изменилось относительно `day_10`

В `day_10` память в основном воспринималась как стратегия работы с историей диалога:

- без сжатия;
- summary;
- sliding window;
- sticky facts;
- branching.

То есть основной вопрос был: **как подготовить short-term контекст для модели**.

В `day_12` поверх этого появилась **многослойная memory model**, где у памяти появился явный смысл:

- `short-term` — то, что стратегия считает текущим краткосрочным контекстом;
- `working` — данные текущей задачи;
- `long-term` — устойчивые предпочтения, договорённости и повторно полезные знания;
- `pending` — кандидаты на сохранение, которые ещё не подтверждены.

Именно это и стало главным развитием проекта: память перестала быть просто “историей диалога” и стала отдельной управляемой подсистемой.

## Что появилось нового

### 1. Layered memory model

Память теперь разделена на отдельные слои:

- `short-term`
- `working`
- `long-term`
- `pending`

Они хранятся раздельно и живут по разным правилам.

### 2. Явное распределение данных по слоям

Теперь сообщение не просто попадает в общую историю.

После пользовательского ввода система:

1. добавляет сообщение в `short-term`;
2. пытается извлечь кандидатов для `working` и `long-term`;
3. валидирует их структурно;
4. решает, что можно применить сразу, а что должно уйти в `pending`;
5. только после этого обновляет долговременные слои памяти.

Это и есть реализация требования:

> вы явно выбирали, что и куда сохраняется

Причём “явность” появилась в двух формах:

- на уровне архитектуры — через отдельные memory layers и pipeline извлечения;
- на уровне UX — через `pending`, подтверждение и ручное управление памятью.

### 3. Pending-flow

Долговременные и спорные записи теперь не обязаны сохраняться автоматически.

Вместо этого появился слой `pending`, где хранятся кандидаты на сохранение.

Пользователь может:

- посмотреть кандидатов;
- подтвердить их;
- отклонить;
- исправить до сохранения.

Это сильно изменило поведение системы:

- память стала менее “магической”;
- спорные извлечения перестали молча попадать в durable memory;
- у пользователя появился реальный контроль над тем, что считается памятью.

### 4. Ручное управление `working` и `long-term`

Поверх pending-flow добавлены прямые команды управления памятью.

Теперь пользователь может вручную:

- добавить запись;
- изменить запись;
- удалить запись.

Это особенно важно, потому что allocator и LLM-извлечение не всегда идеальны.

В результате память перестала зависеть только от автоматического извлечения и стала управляемой вручную.

### 5. Отдельный utility LLM для allocator'а

Для распределения заметок по durable memory был добавлен `LlmMemoryLayerAllocator`, который может использовать отдельную, более дешёвую модель, независимо от основной модели диалога.

Это позволило разделить:

- основную модель ответа;
- утилитную модель для memory extraction.

Если utility LLM недоступен, сохраняется fallback на rule-based allocator.

### 6. Диагностика allocator'а

Чтобы понять, почему кандидаты иногда не извлекаются, добавлен trace-лог:

- [build/logs/llm-memory-layer-allocator.log](C:/Users/compadre/Downloads/Projects/AiAdvent/day_12/build/logs/llm-memory-layer-allocator.log)

В нём видно:

- что пришло на вход allocator'у;
- какие prompt'ы ушли в utility LLM;
- что модель вернула;
- какие кандидаты были извлечены;
- был ли fallback на rule-based allocator.

Это важно не только для отладки, но и для понимания того, как реально формируется память.

## Как теперь устроена память

### Short-term

`short-term` по-прежнему зависит от выбранной стратегии памяти, но теперь смысл слоя стал чётче.

Внутри short-term хранится:

- сырой журнал переписки;
- derived short-term представление, вычисленное стратегией;
- strategy-specific state.

При смене стратегии derived short-term пересобирается заново под новую стратегию.

То есть short-term теперь можно понимать так:

- raw log — источник истины;
- strategy-derived view — то, что реально используется как краткосрочный контекст.

### Working

`working` хранит всё, что относится к текущей задаче:

- цель;
- ограничения;
- срок;
- бюджет;
- интеграции;
- решения;
- открытые вопросы.

Это слой оперативного контекста задачи.

### Long-term

`long-term` хранит устойчивые данные:

- стиль общения;
- постоянные предпочтения;
- архитектурные договорённости;
- повторно полезные знания.

Важно, что этот слой больше не должен засоряться временными деталями текущего шага.

### Pending

`pending` — промежуточный workflow-слой.

Он не является “настоящей памятью” сам по себе, но именно через него проходит подтверждение новых durable notes.

## Как это влияет на ответы ассистента

Теперь итоговый prompt собирается из разных слоёв памяти, а не только из истории диалога.

Высокоуровнево порядок такой:

1. system prompt
2. `long-term`
3. `working`
4. strategy-derived `short-term`

Это изменило поведение агента:

- стиль ответа может поддерживаться через `long-term`;
- данные проекта и текущей задачи могут поддерживаться через `working`;
- краткосрочный диалоговый контекст по-прежнему обеспечивается `short-term`.

То есть ответ модели теперь зависит не только от последней истории диалога, но и от того, **что именно подтверждено и сохранено в слоях памяти**.

## Как данные проходят через новую memory model

Ниже тот же взгляд “по шагам”, но уже с упором именно на изменения `day_12`: layered memory, pending-flow, ручное управление и utility allocator.

### 1. Пользователь вводит строку в CLI

CLI сначала решает, что это:

- встроенная команда с префиксом `/`;
- или обычное сообщение для модели.

Это важно, потому что после `day_12` памятью теперь можно управлять прямо из CLI, а не только через автоматическое извлечение.

### 2. Если это команда, она уходит в CLI-слой, а не в LLM

Команды вроде:

- `/memory`
- `/memory pending`
- `/memory approve`
- `/memory add ...`
- `/memory note edit ...`

обрабатываются внутри CLI и memory manager без запроса к основной модели.

Именно так пользователь получил прямой контроль над слоями памяти.

### 3. Если это обычное сообщение, агент сначала считает локальный preview токенов

Перед основным запросом агент оценивает:

- размер текущего сообщения;
- размер истории;
- размер полного запроса.

На этом этапе данные ещё не попали ни в `working`, ни в `long-term`.

### 4. Сообщение попадает в short-term

Дальше `DefaultMemoryManager` добавляет сообщение в short-term state.

Внутри short-term есть:

- raw log;
- derived short-term view;
- strategy-specific state.

То есть сообщение сначала становится частью краткосрочной памяти, и только потом система решает, нужно ли что-то сохранять в durable memory.

### 5. Запускается allocator для durable memory

После short-term шага менеджер памяти пытается извлечь кандидатов для:

- `working`
- `long-term`

Здесь работают:

- `LlmMemoryLayerAllocator`, если доступна utility LLM;
- `RuleBasedMemoryLayerAllocator`, если utility LLM недоступна или произошёл fallback.

Это ключевое отличие от `day_10`: теперь система не просто ведёт историю, а пытается разложить смысл по отдельным memory layers.

### 6. Structural validator проверяет кандидатов

После allocator’а кандидаты проходят через `MemoryCandidateValidator`.

Сейчас он делает только структурную проверку:

- допустим ли слой;
- допустима ли категория;
- не пустой ли текст.

Он больше не принимает смысловых решений за пользователя и не отбрасывает кандидатов по category-specific regex.

### 7. Confirmation policy делит кандидатов на auto-apply и pending

Дальше `MemoryConfirmationPolicy` решает:

- что можно сразу сохранить;
- что нужно отправить в `pending`.

Именно здесь появляется явный шаг подтверждения перед durable memory.

### 8. Auto-apply обновляет working и long-term

Кандидаты из `auto-apply` сразу применяются к нужному слою:

- в `working`;
- или в `long-term`.

То есть часть памяти может обновляться автоматически, но только после отдельного этапа классификации.

### 9. Pending-кандидаты сохраняются отдельно

Если кандидат требует подтверждения, он не теряется и не смешивается с настоящей памятью.

Он попадает в отдельный слой:

- `pending`

Дальше пользователь уже может:

- посмотреть его;
- подтвердить;
- отклонить;
- исправить.

Это одно из главных архитектурных изменений `day_12`.

### 10. Пользователь может вручную управлять durable memory

Помимо pending-flow, теперь есть прямое управление `working` и `long-term`.

Пользователь может:

- добавить запись;
- изменить запись;
- удалить запись.

Для этого у заметок появились устойчивые `id`.

Это означает, что память теперь формируется не только allocator’ом, но и прямыми пользовательскими действиями.

### 11. Short-term стратегия пересчитывает derived view

После обработки durable memory short-term стратегия пересобирает derived short-term представление.

Это значит:

- raw log остаётся источником истины;
- derived short-term view пересчитывается в соответствии с выбранной стратегией.

При смене стратегии derived short-term строится заново.

### 12. Из memory layers собирается итоговый prompt

Когда слои памяти уже обновлены, система собирает итоговый контекст для модели.

Высокоуровнево туда входят:

1. system prompt
2. `long-term`
3. `working`
4. derived `short-term`

То есть модель отвечает уже не только на основе истории, а на основе layered memory model.

### 13. Основная модель генерирует ответ

После сборки prompt идёт запрос к основной диалоговой модели.

На этом этапе уже видно, как новые memory layers влияют на ответ:

- `long-term` влияет на стиль и устойчивые правила;
- `working` влияет на содержание текущей задачи;
- `short-term` даёт ближайший диалоговый контекст.

### 14. Ответ ассистента показывается пользователю

После ответа модельный результат выводится в CLI.

Если в процессе появились новые pending-кандидаты, пользователь получает явную подсказку, что их можно посмотреть и обработать отдельно.

### 15. Если allocator использовал utility LLM, его работа попадает в trace-лог

Для диагностики `LlmMemoryLayerAllocator` пишет trace в файл:

- [build/logs/llm-memory-layer-allocator.log](C:/Users/compadre/Downloads/Projects/AiAdvent/day_12/build/logs/llm-memory-layer-allocator.log)

Там можно увидеть:

- входное сообщение;
- prompt для utility LLM;
- сырой ответ;
- распарсенные заметки;
- итоговые кандидаты;
- fallback на rule-based allocator, если он произошёл.

Это делает новую memory subsystem не только управляемой, но и наблюдаемой.

## Как теперь выглядит пользовательское управление памятью

Все встроенные команды CLI теперь начинаются с `/`.

Это было сделано специально, чтобы:

- сократить вероятность случайной отправки команды в LLM;
- сделать команды и обычные сообщения визуально и семантически разными.

Основные команды памяти:

- `/memory`
- `/memory short`
- `/memory working`
- `/memory long`
- `/memory pending`
- `/memory approve [id...]`
- `/memory reject [id...]`
- `/memory edit <id> text|layer|category <value>`
- `/memory add working <category> <text>`
- `/memory add long <category> <text>`
- `/memory note edit <working|long> <id> text|category <value>`
- `/memory note delete <working|long> <id>`

Из-за этого memory model стала не только внутренней архитектурой, но и частью пользовательского интерфейса.

Главное развитие `day_12` по сравнению с `day_10` можно сформулировать так:

Появились:

- разные слои памяти с разным смыслом;

- trace-диагностика allocator'а.

То есть проект стал ближе не просто к “чату с памятью”, а к **ассистенту с управляемой и наблюдаемой memory subsystem**.
