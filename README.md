## Что умеет проект сейчас

### Memory subsystem

- layered memory:
  - `short-term`
  - `working`
  - `long-term`
  - `pending`
- short-term стратегии памяти;
- выделение кандидатов в durable memory;
- pending-flow с подтверждением;
- ручное редактирование памяти;
- multi-user профили и `active user`.

### Task subsystem

- формализованное состояние задачи:
  - `stage`
  - `currentStep`
  - `expectedAction`
  - `status`
- pause / resume / done;
- persistence задач между перезапусками;
- несколько задач в одной сессии;
- одна активная задача в каждый момент времени;
- только активная задача влияет на prompt и orchestration.

### Prompt / Orchestration

- memory и tasks собираются как отдельные prompt contributions;
- финальный `system message` формируется в одном месте;
- агент учитывает `task state` при выборе режима ответа;
- при `paused` задача не продолжается автоматически;
- при смене активной задачи в prompt уходит именно она.

### Smoke / Debug

- scripted smoke-check для memory и task flow;
- structured trace через `DebugTraceListener`;
- transcript собирается из trace, а не из парсинга CLI-строк.

## Как устроена memory subsystem

### Short-term

`short-term` хранит:

- raw log диалога;
- derived short-term view;
- strategy-specific state.

Raw log остаётся источником истины, а derived view зависит от выбранной стратегии.

### Working

`working` хранит оперативный контекст:

- текущие цели;
- ограничения;
- решения;
- открытые вопросы;
- данные текущей работы.

### Long-term

`long-term` хранит устойчивые знания:

- предпочтения;
- договорённости;
- повторно полезные решения;
- проектные правила;
- user-scoped заметки профиля.

### Pending

`pending` хранит кандидатов на сохранение, которые ещё не подтверждены пользователем.

## Pipeline, привязанный к пользовательскому опыту

Этот блок описывает не только технический data flow, но и то, как пользователь реально взаимодействует с проектом после доработок task subsystem. Ниже каждый шаг показан сразу в трёх плоскостях: что делает пользователь, какие компоненты на это реагируют и как это влияет на состояние, prompt и итоговый ответ агента.

### 1. Пользователь начинает новый рабочий трек

Пользователь создаёт задачу через `/task start <title>` или меняет её поля через `/task stage`, `/task step`, `/task expect`.

На это реагируют:

- CLI parser и session controller;
- `TaskManager`;
- persistence слой task subsystem.

Что происходит внутри:

- команда разбирается в CLI;
- `TaskManager` обновляет `TaskSessionState`;
- если активная задача уже была, она переводится в `paused`;
- новая или изменённая задача сохраняется в `config/tasks/`.

Что это даёт пользователю:

- появляется явное состояние задачи, а не “контекст только в голове модели”;
- система запоминает, над чем пользователь работает прямо сейчас;
- при следующем сообщении агент получает уже структурированный task context.

### 2. Пользователь переключается между задачами

Пользователь вызывает `/task switch <id>` или `/task resume <id>`.

На это реагируют:

- `TaskManager`;
- `TaskSessionState`;
- CLI renderer.

Что происходит внутри:

- меняется `activeTaskId`;
- предыдущая активная задача уходит в `paused`, если это нужно;
- завершённая задача не может стать активной через обычное переключение;
- в session state остаётся несколько задач, но активной считается только одна.

Что это даёт пользователю:

- можно держать несколько рабочих треков параллельно;
- при возврате к задаче не нужно заново объяснять весь контекст;
- следующая реплика в LLM уже пойдёт с новой активной задачей.

### 3. Пользователь отправляет обычное сообщение агенту

Это уже не CLI-команда, а обычный запрос в модель.

На этом шаге начинают работать обе подсистемы:

- `MemoryManager`;
- `TaskManager`;
- `TaskOrchestrationService`;
- `AgentPromptAssembler`;
- `MrAgent`.

Что происходит внутри:

1. CLI передаёт обычное сообщение агенту.
2. `MemoryManager` обновляет `short-term`, перерассчитывает memory state и готовит свой prompt context.
3. `TaskManager` отдаёт prompt context только для активной задачи.
4. `TaskOrchestrationService` анализирует активную задачу и решает:
   - нужно ли блокировать продолжение;
   - нужен ли дополнительный guidance;
   - или можно отвечать без специальных ограничений.
5. `AgentPromptAssembler` централизованно собирает финальный `system message` из:
   - базового system prompt;
   - memory contribution;
   - task contribution;
   - orchestration guidance.

Что это даёт пользователю:

- память и задачи работают вместе, но не смешиваются в одну неявную сущность;
- агент отвечает в контексте текущей активной задачи;
- при переключении задачи меняется не только текст в `/task show`, но и реальный контекст ответа модели.

### 4. Memory subsystem подмешивает устойчивый и текущий контекст

Пока пользователь пишет обычное сообщение, memory subsystem отдельно собирает свой вклад.

На этом шаге важны:

- `short-term` как ближайшая история диалога;
- `working` как оперативный контекст;
- `long-term` как устойчивые знания;
- `pending` как кандидаты на сохранение;
- `active user` как источник персонализации.

Что происходит внутри:

- сообщение сначала попадает в `short-term`;
- allocator пытается извлечь кандидатов для durable memory;
- кандидаты валидируются и либо применяются, либо уходят в `pending`;
- memory subsystem формирует effective messages и memory prompt contribution.

Что это даёт пользователю:

- агент учитывает не только текущую задачу, но и накопленные знания проекта;
- персонализация и memory layers продолжают работать поверх task subsystem;
- durable memory и task state остаются разными сущностями с разной ответственностью.

### 5. Task subsystem задаёт рабочий режим ответа

Активная задача влияет не только на содержимое prompt, но и на поведение агента.

На этом шаге ключевые поля:

- `status`
- `expectedAction`
- `stage`

Что происходит внутри:

- `status` определяет, можно ли продолжать рабочий трек;
- `expectedAction` подсказывает, какого шага не хватает для продвижения;
- `stage` задаёт режим ответа;
- `TaskOrchestrationService` формирует policy:
  - `block`
  - `guide`
  - `none`

Примеры:

- `paused` блокирует продолжение задачи без явного возобновления;
- `planning + user_input` подталкивает модель к уточнениям и просьбе о недостающих решениях;
- `execution` помогает обсуждать текущую реализацию;
- `completion` помогает подвести итог, не теряя текущее состояние задачи.

Что это даёт пользователю:

- агент начинает вести себя по-разному в зависимости от состояния задачи;
- уменьшается необходимость заново объяснять “на каком мы этапе”;
- task state становится реальной рабочей опорой, а не просто заметкой в prompt.

### 6. Финальный prompt собирается централизованно

Это важный архитектурный эффект последних доработок.

На этом шаге:

- memory subsystem не правит `system message` сама;
- task subsystem не правит `system message` сама;
- финальная сборка живёт в одном месте.

Что происходит внутри:

- `AgentPromptAssembler` получает отдельные contributions;
- собирает единый final prompt;
- следит, чтобы task state был приоритетнее исторических формулировок там, где это важно;
- один и тот же подход используется и для реального запроса, и для debug/preview flow.

Что это даёт пользователю:

- поведение агента становится более предсказуемым;
- task context не теряется среди истории;
- проще диагностировать, почему модель ответила именно так.

### 7. Пользователь видит не только ответ, но и управляемый рабочий flow

После ответа агент показывает не просто текст модели, а уже результат работы нескольких связанных подсистем.

Пользовательский эффект:

- можно вести несколько задач параллельно;
- можно вернуться к задаче без повторного ввода контекста;
- можно ставить задачу на паузу, завершать и возобновлять;
- активная задача реально влияет на содержание и тон ответа модели;
- память продолжает поддерживать долгоживущий контекст проекта и пользователя.

Именно это и есть главный смысл текущих доработок: task subsystem стала частью пользовательского опыта, а не только внутренней структуры данных.

## Как устроена task subsystem

У каждой задачи есть:

- `stage` — текущий этап работы;
- `currentStep` — текущий конкретный шаг;
- `expectedAction` — какое действие нужно для продвижения;
- `status` — жизненное состояние задачи.

Базовые этапы:

- `planning`
- `execution`
- `validation`
- `completion`

Базовые статусы:

- `active`
- `paused`
- `done`

Дополнительно task subsystem хранит `TaskSessionState`:

- список задач;
- `activeTaskId`;
- правило “несколько задач, одна активная”.

Если создаётся новая задача, предыдущая активная уходит в `paused`.  
Завершённая задача не может стать активной снова через обычный `switch`.

## Команды задач

Сейчас доступны:

- `/task show`
- `/task list`
- `/task start <title>`
- `/task switch <id>`
- `/task resume <id>`
- `/task done <id>`
- `/task pause`
- `/task stage <planning|execution|validation|completion>`
- `/task step <text>`
- `/task expect <user_input|agent_execution|user_confirmation|none>`

Минимальная модель использования такая:

1. создать задачу;
2. зафиксировать этап и текущий шаг;
3. при необходимости поставить на паузу;
4. переключиться на другую задачу;
5. позже вернуться через `switch` или `resume`;
6. завершить через `done`.

## Как task state влияет на ответы

Агент использует task state в двух формах:

1. как prompt context;
2. как orchestration policy.

Это даёт следующий эффект:

- `paused` блокирует продолжение рабочего трека;
- `expectedAction` подсказывает, чего сейчас не хватает;
- `stage` задаёт режим ответа;
- история диалога используется как фон, но не как главный источник истины о текущей задаче.

Только активная задача влияет на prompt и orchestration. Остальные задачи хранятся в session state, но не подмешиваются в ответ напрямую.

## План развития task manager

Текущее состояние уже покрывает базовый рабочий flow, но дальше task subsystem можно развивать в нескольких направлениях:

- сделать UX списка задач более удобным и менее “техническим”;
- добавить user-facing сводки при `switch/resume`;
- добавить agent suggestions для перевода задачи между этапами;
- аккуратно усилить помощь агенту при создании новой задачи из обычного диалога;
- вынести некоторые debug и token-counting детали выше memory manager;
- расширить smoke-check новыми сценариями по мере роста task UX.
