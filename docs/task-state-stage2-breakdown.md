# Task State Stage 2 Breakdown

Этот документ детализирует второй этап из [task-state-roadmap.md](./task-state-roadmap.md) в виде крупных блоков работ.

Граница этапа 2:

- всё ещё поддерживается только одна текущая задача;
- task state по-прежнему меняется только через `/task`-команды;
- агент начинает учитывать `stage`, `status` и `expectedAction` в своём поведении;
- строгая валидация переходов между stage всё ещё не требуется;
- несколько задач, suggestions и multitask UX не входят в scope этапа 2.

## Блок 1. Behavior Contract

### Цель блока

Ясно зафиксировать, как именно task state должен влиять на поведение агента на этапе 2.

### Подзадача 1.1. Зафиксировать поведенческий контракт stage/status/expectedAction

#### ТЗ

Нужно определить минимальные правила, которые агент обязан уважать в ответах.

Обязательные правила:

- если `status = PAUSED`, агент не продолжает рабочий трек автоматически;
- если `expectedAction = USER_INPUT`, агент явно показывает, что ждёт решение пользователя;
- если `stage = PLANNING`, агент не перепрыгивает сразу в реализацию без явного перехода;
- если `stage = VALIDATION`, агент отвечает в режиме проверки;
- если `stage = COMPLETION`, агент помогает подвести итог, оформить результат или закрыть задачу;
- если активной задачи нет, агент не должен притворяться, что task workflow всё ещё существует.

#### Результат

Есть единый контракт поведения, на который можно опираться в orchestration-коде, тестах и smoke-check.

#### Готово, когда

- правила записаны в одном месте;
- нет двусмысленности между `stage`, `status` и `expectedAction`;
- из контракта понятно, что именно enforce'ится кодом, а что задаётся через prompt.

### Подзадача 1.2. Развести жёсткие и мягкие ограничения

#### ТЗ

Нужно разделить поведение на два класса:

- жёсткие ограничения, которые код обязан enforce'ить;
- мягкие подсказки, которые можно задавать через prompt/context.

На этапе 2 рекомендуется:

- жёстко enforce'ить `PAUSED`;
- жёстко учитывать отсутствие активной задачи, если это влияет на workflow;
- мягко направлять поведение по `PLANNING`, `VALIDATION`, `COMPLETION`;
- мягко интерпретировать `ExpectedAction`, кроме явного случая `USER_INPUT`.

#### Результат

Становится понятно, какие сценарии должны блокироваться на уровне orchestration, а какие можно оставить на уровне поведенческих подсказок.

#### Готово, когда

- список жёстких и мягких правил зафиксирован;
- код не пытается слишком рано превратить этап 2 в строгий state machine;
- граница между orchestration и prompt остаётся простой.

## Блок 2. Task Orchestration

### Цель блока

Добавить слой, который читает task state перед ответом агента и управляет поведением без смешения task и memory подсистем.

### Подзадача 2.1. Спроектировать orchestration-абстракцию

#### ТЗ

Нужно ввести отдельную сущность, например:

- `TaskStateGuard`
- `TaskAwareResponsePolicy`
- `TaskOrchestrationService`

Ответственность:

- читать текущий `TaskPromptContext` или `TaskState`;
- определять, какие ограничения или подсказки действуют прямо сейчас;
- возвращать структурированный результат для агента.

Предпочтительный выход:

- не просто строка;
- а typed-результат вроде `TaskGuardDecision` / `TaskBehaviorHint`.

#### Результат

Поведение агента становится управляемым отдельным orchestration-слоем, а не размазывается по `MrAgent`.

#### Готово, когда

- новая абстракция существует;
- её ответственность ограничена task-aware поведением;
- она не знает про storage и не модифицирует task state сама по себе.

### Подзадача 2.2. Реализовать typed-модель решений orchestration

#### ТЗ

Нужно создать модели, которыми orchestration-слой будет описывать поведение.

Примерно:

- `TaskGuardDecision`
- `TaskBehaviorHint`
- `TaskBehaviorMode`

Минимально модель должна уметь выразить:

- workflow заблокирован из-за `PAUSED`;
- сейчас ожидается пользовательский ввод;
- агенту нужно отвечать в режиме planning;
- агенту нужно отвечать в режиме validation;
- агенту нужно отвечать в режиме completion;
- специальных ограничений нет.

#### Результат

Агент получает не raw string-магии, а typed-подсказки для поведения.

#### Готово, когда

- все обязательные режимы этапа 2 выражаются через typed-модель;
- `MrAgent` может использовать эту модель без знания деталей task logic;
- новые режимы можно добавить без переделки старого API.

## Блок 3. Agent Integration

### Цель блока

Подключить orchestration-слой к реальному agent flow и добиться заметного изменения поведения.

### Подзадача 3.1. Подключить task-aware behavior в `MrAgent`

#### ТЗ

Нужно встроить orchestration-решение в основной путь ответа агента.

Требования:

- использовать уже существующую prompt assembly архитектуру;
- не возвращаться к редактированию system prompt из memory/task подсистем;
- поведение `ask()` и `previewTokenStats()` должно оставаться согласованным;
- `MrAgent` должен остаться orchestration-слоем, а не хранить всю task-логику внутри себя.

#### Результат

При одинаковом пользовательском сообщении агент отвечает по-разному в зависимости от текущего task state.

#### Готово, когда

- `PAUSED` реально останавливает автоматическое продолжение workflow;
- `USER_INPUT` влияет на тон и направление ответа;
- `PLANNING`, `VALIDATION`, `COMPLETION` реально ощущаются в ответах.

### Подзадача 3.2. Сохранить разделение task и memory

#### ТЗ

Нужно при интеграции проверить, что архитектурные границы не нарушены.

Нельзя:

- переносить task state в memory notes;
- использовать `pending` для task orchestration;
- делать memory subsystem ответственной за task-aware response behavior.

Нужно:

- читать memory context и task context отдельно;
- соединять их только на orchestration-уровне агента.

#### Результат

Task-aware поведение добавлено без архитектурного регресса.

#### Готово, когда

- memory и task подсистемы остаются независимыми;
- orchestration-слой использует обе, но не смешивает их модели;
- в коде нет новых shortcut'ов через memory state для task поведения.

## Блок 4. CLI / UX Feedback

### Цель блока

Сделать так, чтобы пользователь явно видел, что агент учитывает task state, а не просто имеет внутренние флаги.

### Подзадача 4.1. Сделать ответы агента понятными для paused/user_input сценариев

#### ТЗ

Нужно обеспечить понятный UX для ключевых случаев этапа 2.

Примеры ожидаемого поведения:

- при `PAUSED` агент сообщает, что текущая задача на паузе;
- при `USER_INPUT` агент сообщает, что сейчас ждёт решение пользователя;
- при `COMPLETION` агент помогает оформить итог, а не просто продолжает выполнение как обычно.

Требования:

- не добавлять suggestions как отдельный feature;
- не менять task state автоматически;
- не требовать от пользователя помнить внутреннюю модель подсистемы.

#### Результат

Поведение становится заметным и объяснимым с пользовательской точки зрения.

#### Готово, когда

- агент не выглядит “сломавшимся” в paused-сценарии;
- причина ожидания user input явно понятна;
- completion-сценарий выглядит как завершение, а не как обычная генерация.

### Подзадача 4.2. При необходимости расширить smoke-check этапа 2

#### ТЗ

Нужно подготовить следующую версию smoke-check для stage 2.

Минимальные сценарии:

1. задача в `planning`, пользователь просит “давай писать код”;
2. задача в `paused`, пользователь просит “продолжай”;
3. `expectedAction = user_input`, пользователь задаёт общий вопрос “что дальше?”;
4. задача в `completion`, пользователь просит завершить работу.

#### Результат

Появляется практическая проверка того, что task-aware behavior реально работает, а не только покрыт unit-тестами.

#### Готово, когда

- smoke-сценарии отражают ключевые режимы этапа 2;
- scripted/manual check проверяет именно поведение агента, а не только команды `/task`.

## Блок 5. Verification

### Цель блока

Подтвердить, что этап 2 работает на уровне доменной логики, agent behavior и пользовательского сценария.

### Подзадача 5.1. Добавить unit и integration tests на orchestration

#### ТЗ

Нужно покрыть тестами:

- `PAUSED` блокирует продолжение workflow;
- `USER_INPUT` отражается в поведении;
- `PLANNING` не ведёт к немедленному выполнению без явного перехода;
- `VALIDATION` даёт режим проверки;
- `COMPLETION` даёт режим подведения итогов;
- отсутствие активной задачи обрабатывается корректно.

Требования:

- тесты должны проверять именно поведенческий контракт этапа 2;
- multitask-сценарии в эти тесты не входят;
- строгая transition validation не должна внезапно появиться через тесты.

#### Результат

Task-aware behavior защищён от регрессий.

#### Готово, когда

- все ключевые режимы покрыты;
- тесты читаемо описывают, что именно считается корректным поведением;
- изменения в orchestration-коде быстро проверяются локально.

### Подзадача 5.2. Пройти ручной сценарий этапа 2

#### ТЗ

Нужно прогнать ручной сценарий:

1. `/task start Реализовать task behavior`
2. `/task stage planning`
3. обычное сообщение: `Давай уже писать код`
4. убедиться, что агент не игнорирует planning
5. `/task expect user_input`
6. обычное сообщение: `Что дальше?`
7. убедиться, что агент показывает ожидание пользовательского решения
8. `/task pause`
9. обычное сообщение: `Продолжай`
10. убедиться, что paused-задача не продолжается автоматически
11. `/task resume`
12. `/task stage validation`
13. обычное сообщение про проверку результата
14. убедиться, что ответ выглядит как validation
15. `/task stage completion`
16. обычное сообщение: `Подведи итог`
17. убедиться, что агент помогает завершить задачу

#### Результат

Есть реальное подтверждение, что этап 2 заметен пользователю, а не только существует в коде.

#### Готово, когда

- сценарий проходит без ручных обходов;
- поведение отличается в зависимости от task state;
- пользовательские сообщения без `/task` начинают учитываться task-aware образом, но не меняют task state автоматически.

## Рекомендуемый порядок реализации

1. Блок 1. Behavior Contract
2. Блок 2. Task Orchestration
3. Блок 3. Agent Integration
4. Блок 4. CLI / UX Feedback
5. Блок 5. Verification

## Что не делаем в этапе 2

- несколько задач;
- `activeTaskId` и multitask UX;
- suggestions от агента;
- user-scoped tasks;
- жёсткую валидацию переходов между stage;
- автоматическое изменение task state из обычных пользовательских сообщений;
- автосоздание новых задач по смыслу сообщения пользователя.
