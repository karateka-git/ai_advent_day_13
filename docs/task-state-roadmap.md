# Task State Roadmap

## Цель

Прийти к модели, где в чате можно вести несколько задач, при этом активна только одна. У каждой задачи есть формализованное состояние:

- этап задачи
- текущий шаг
- ожидаемое действие
- статус задачи

Ключевые свойства системы:

- на ранних этапах task state меняется только через явные команды `/task ...`;
- агент обязан учитывать task state в своём поведении;
- `working memory` и task state остаются разными подсистемами;
- `pending` остаётся механизмом memory subsystem и не используется для переходов task state;
- на ранних этапах переходы между stage не валидируются жёстко: важнее зафиксировать типизированное состояние и влияние этого состояния на поведение агента;
- на поздних этапах агент может предлагать изменения task state, но не применять их без подтверждения пользователя.

Рабочее определение задачи:

- Задача — это отдельный рабочий трек внутри чата, который пользователь может явно начать, приостановить, возобновить или завершить.

Временная политика scope:

- на первых этапах task list считается `conversation-scoped`;
- переключение активного пользователя не меняет task list автоматически;
- влияние `active user` ограничивается профилем и memory subsystem;
- возможный переход к `user-scoped tasks` рассматривается как отдельное будущее архитектурное решение.

## Общие архитектурные решения

- Task subsystem живёт отдельно от Memory subsystem.
- `agent.memory.*` отвечает за память и заметки.
- `agent.task.*` отвечает за состояние задач.
- `working memory` не является task state.
- task state не хранится как обычные memory notes.
- `pending` не участвует в task transitions.
- На ранних этапах источник изменений task state: только пользователь через команды.
- На ранних этапах `stage` типизирован, но переходы между stage остаются свободными и не ограничиваются жёсткой state-transition валидацией.
- На первых этапах task list хранится вместе с состоянием текущего диалога и восстанавливается вместе с этой сессией.
- Переключение `active user` не должно автоматически переключать или очищать task list.
- На поздних этапах агент может только предлагать изменения task state.

Предлагаемая структура пакетов:

- `agent.task.model`
- `agent.task.core`
- `agent.task.persistence`
- `agent.task.prompt`

Подход к моделированию stage:

- используем `B-lite` модель;
- на старте `stage` остаётся типизированным через enum;
- поверх enum сразу вводится слой definitions/metadata;
- richer-стартовый набор stage: `planning / execution / validation / completion`;
- завершённость задачи хранится в `status`, а не в `stage`;
- набор stage трактуется как стартовый, а не как окончательный и единственно возможный.

## Этап 1. Одна активная задача как отдельная подсистема

### Смысл этапа

Вводим формализованное состояние одной текущей задачи, не смешивая его с memory.

### Что видит пользователь

У пользователя появляется одна текущая задача, у которой есть:

- `title`
- `stage`
- `current step`
- `expected action`
- `status`

Управление идёт через команды, например:

```text
/task start Реализовать state machine
/task show
/task stage planning
/task step Продумать модель TaskState
/task expect user_input
/task pause
/task resume
/task done
```

Обычные сообщения task state не меняют.

### Что пользователь может делать

- завести текущую задачу;
- посмотреть её состояние;
- обновить этап;
- обновить текущий шаг;
- указать ожидаемое действие;
- поставить задачу на паузу;
- снять задачу с паузы;
- завершить задачу;
- очистить текущую задачу, если понадобится.

### Что это даёт пользователю

- появляется явная рамка текущей работы;
- можно вернуться к задаче позже без повторного объяснения;
- task state не смешивается с проектными заметками и памятью.

### Технические изменения

Новые доменные модели:

- `TaskStage`
- `TaskStageDefinition`
- `TaskStages`
- `TaskStatus`
- `ExpectedAction`
- `TaskState`

Примерный состав:

- `TaskStage`: `PLANNING`, `EXECUTION`, `VALIDATION`, `COMPLETION`
- `TaskStageDefinition`: metadata для отображения и смысловой трактовки stage
- `TaskStages`: единый каталог стартовых stage definitions
- `TaskStatus`: `ACTIVE`, `PAUSED`, `DONE`
- `ExpectedAction`: `USER_INPUT`, `AGENT_EXECUTION`, `USER_CONFIRMATION`, `NONE`

Идея `B-lite` на старте:

- `TaskStage` хранит типобезопасное значение;
- `TaskStageDefinition` хранит `label`, `description` и при необходимости дополнительную metadata для UI и orchestration;
- логика отображения и дальнейшего расширения stage не размазывается по `when`-веткам по всему проекту, а концентрируется в одном каталоге.
- `expected action` хранится рядом со `stage` как отдельное поле task state и описывает следующий необходимый шаг для продвижения задачи, а не дублирует сам `stage`.

Новая core-логика:

- `TaskStateMachine`
- `TaskManager` или `DefaultTaskManager`

Что должна уметь core-логика:

- создавать текущую задачу;
- обновлять `stage`;
- обновлять `currentStep`;
- обновлять `expectedAction`;
- выполнять `pause`, `resume`, `done`;
- хранить и менять типизированное состояние задачи без жёсткой валидации переходов между stage на старте.

Новая persistence-абстракция:

- `TaskStateRepository`
- `JsonTaskStateRepository`

Новый prompt-слой:

- `TaskPromptAssembler`
- компактно формирует task context для LLM

Дополнительная инфраструктура stage:

- каталог `TaskStages` используется CLI, prompt assembly и orchestration-слоем как источник подписей и смысловых описаний stage;
- это упрощает будущий переход от жёсткого enum к более гибкой модели, если он понадобится.

Политика раннего поведения:

- если пользователь пишет обычное сообщение без `/task`, task state не меняется;
- если текущая задача `paused`, обычное сообщение не возобновляет её автоматически;
- если пользователь хочет начать новую задачу при наличии активной, это должно происходить через явное действие пользователя.

CLI-слой:

- новые команды `/task`
- parser/model/controller/rendering/help

### Вероятные места интеграции

Новые файлы:

- `src/main/kotlin/agent/task/model/*`
- `src/main/kotlin/agent/task/core/*`
- `src/main/kotlin/agent/task/persistence/*`
- `src/main/kotlin/agent/task/prompt/*`

Изменяемые файлы:

- `src/main/kotlin/agent/core/Agent.kt`
- `src/main/kotlin/agent/impl/MrAgent.kt`
- `src/main/kotlin/bootstrap/AgentFactory.kt`
- `src/main/kotlin/ui/cli/CliCommandParser.kt`
- `src/main/kotlin/ui/cli/CliCommands.kt`
- `src/main/kotlin/ui/cli/CliSessionController.kt`
- `src/main/kotlin/ui/cli/CliRenderer.kt`

### Как понять, что этап готов

Функционально:

- можно создать одну задачу;
- можно просмотреть её состояние;
- можно менять `stage`, `step`, `expected action`, `status`;
- состояние сохраняется между сессиями;
- обычные сообщения task state не ломают.

Что проверить вручную:

1. Создать задачу.
2. Выставить `planning`, `step`, `expected action`.
3. Перезапустить приложение.
4. Проверить, что состояние сохранилось.
5. Поставить задачу на паузу и снять с паузы.
6. Завершить задачу.

Какие тесты нужны:

- создание задачи;
- pause/resume;
- done;
- save/load состояния.

## Этап 2. Агент начинает подчиняться task state

### Смысл этапа

Task state становится не просто сохранёнными метаданными, а реально влияет на поведение агента.

### Что видит пользователь

Пользователь по-прежнему управляет состоянием через `/task`, но теперь после изменения task state поведение агента реально меняется.

Примеры:

- если задача в `planning`, агент не прыгает сразу в реализацию;
- если задача в `validation`, агент отвечает в режиме проверки;
- если задача в `completion`, агент помогает подвести итог, оформить результат или закрыть задачу;
- если задача `paused`, агент не продолжает этот рабочий трек автоматически;
- если `expected action = user_input`, агент показывает, что ждёт решения пользователя.

### Что пользователь может делать

- осознанно переключать режим работы через `stage`;
- явно ставить задачу на паузу;
- явно задавать ожидаемое действие;
- получать ответы, которые уважают текущее состояние задачи.

### Что это даёт пользователю

- task state перестаёт быть декоративным;
- появляется дисциплина процесса;
- уменьшаются случайные скачки между planning, execution и validation.

### Технические изменения

Новая orchestration-абстракция:

- `TaskStateGuard`
- `TaskAwareResponsePolicy`
- `TaskOrchestrationService`

Что она делает:

- анализирует активную задачу перед ответом;
- определяет, есть ли ограничения на поведение;
- подсказывает агенту, как себя вести.

Возможные дополнительные модели:

- `TaskBehaviorHint`
- `TaskGuardDecision`

Подход к интеграции:

- часть правил задаётся через task block в prompt;
- часть правил enforce'ится кодом.
- при интерпретации stage желательно опираться не только на enum, но и на `TaskStageDefinition`, чтобы семантика stage жила централизованно.

На этом этапе особенно имеет смысл enforce'ить кодом:

- `paused` задачу;
- отсутствие активной задачи, если это влияет на поведение.

Важно:

- даже на этом этапе переходы между stage всё ещё могут оставаться свободными;
- агент обязан учитывать текущее состояние задачи, но система пока не обязана запрещать переходы вроде `planning -> completion`.
- `expected action` должен интерпретироваться как ближайшее требуемое действие:
  - `USER_INPUT` — нужен ответ или решение пользователя;
  - `AGENT_EXECUTION` — следующий рабочий ход за агентом;
  - `USER_CONFIRMATION` — нужен confirm или approve от пользователя;
  - `NONE` — явного следующего действия не зафиксировано.
- `expected action` не должен дублировать список stage и не должен использоваться как вторая версия `TaskStage`.

### Вероятные места интеграции

- `src/main/kotlin/agent/impl/MrAgent.kt`
- новый слой `agent.task.core`
- wiring в `src/main/kotlin/bootstrap/AgentFactory.kt`

### Как понять, что этап готов

Функционально:

- task state подмешивается в prompt или orchestration;
- ответы агента различаются в зависимости от `stage/status/expected action`;
- paused-задача не продолжается автоматически.

Что проверить вручную:

1. Завести задачу.
2. Поставить `planning`.
3. Попросить “давай уже писать код”.
4. Проверить, что агент не игнорирует planning.
5. Поставить задачу на паузу.
6. Попросить “продолжай”.
7. Проверить, что агент не продолжает workflow без resume.
8. Поставить `expected action = user_input`.
9. Проверить, что агент ждёт решения, а не подменяет его.
10. Поставить `stage = completion`.
11. Проверить, что агент помогает завершить задачу и собрать итог, а не уходит обратно в обычное выполнение.

Какие тесты нужны:

- paused-задача блокирует продолжение workflow;
- planning/validation отражаются в task context;
- expected action влияет на формируемое поведение.

## Этап 3. Внутренняя модель нескольких задач

### Смысл этапа

Подготовить архитектуру к нескольким задачам, даже если UI пока ещё не полностью multitask-friendly.

### Что видит пользователь

Пользователь начинает новую задачу без ощущения, что старая исчезла. Даже если интерфейс ещё упрощён, система уже хранит несколько задач.

Пример:

```text
/task start Спроектировать task subsystem
/task pause
/task start Проверить persistence
```

### Что пользователь может делать

- заводить новую задачу, не теряя старую;
- хранить paused и done задачи;
- постепенно привыкать, что задач может быть больше одной.

### Что это даёт пользователю

- новая задача перестаёт “стирать” старую;
- появляется фундамент для настоящего pause/resume между задачами.

### Технические изменения

Переход от одной задачи к реестру задач.

Вместо:

- `TaskState?`

Появляется:

- `TaskSessionState`
- `TaskItem`
- `activeTaskId`

Примерно:

- `TaskItem`
  - `id`
  - `title`
  - `stage`
  - `status`
  - `currentStep`
  - `expectedAction`
- `TaskSessionState`
  - `tasks: List<TaskItem>`
  - `activeTaskId: String?`
  - возможно `nextTaskId`

Persistence:

- `TaskSessionStateRepository`
- JSON-репозиторий для всей task session state

Core-логика:

- task manager теперь работает не с одной задачей, а с набором;
- умеет создавать новую без потери старой;
- умеет находить активную задачу.
- уже существующий слой `TaskStages` продолжает быть единым источником metadata для stage у каждой задачи.

Политика multitask на этом этапе и далее:

- новая задача не должна молча уничтожать старую;
- paused-задачи остаются в task list;
- done-задачи остаются в task list как завершённые записи;
- переключение active user не должно автоматически менять active task или task list.

### Вероятные места интеграции

- `agent.task.model/*`
- `agent.task.persistence/*`
- `agent.task.core/*`
- CLI можно пока менять минимально

### Как понять, что этап готов

Функционально:

- задачи хранятся списком;
- активная задача определяется явно;
- старая задача не исчезает после старта новой;
- session state сохраняется и восстанавливается.

Что проверить вручную:

1. Создать задачу A.
2. Поставить её на паузу.
3. Создать задачу B.
4. Перезапустить приложение.
5. Проверить, что обе задачи сохранились и активная восстановилась корректно.

Какие тесты нужны:

- создание второй задачи не удаляет первую;
- активная задача определяется корректно;
- сериализация и десериализация списка задач работает.

## Этап 4. Полноценная модель: несколько задач, одна активная

### Смысл этапа

Довести систему до целевого сценария: несколько задач, но активна только одна.

### Что видит пользователь

Появляются полноценные multitask-команды:

```text
/task list
/task show
/task start Реализовать task state machine
/task pause
/task start Проверить README
/task switch t1
/task resume t1
/task done t2
```

Пользователь может:

- смотреть список задач;
- видеть, какая задача активна;
- переключаться между задачами;
- возобновлять paused-задачу;
- завершать конкретную задачу.

### Что это даёт пользователю

- pause становится по-настоящему полезной;
- можно вести несколько рабочих треков в одном чате;
- возврат к старой задаче происходит без пересказа контекста.

### Технические изменения

Task manager становится полноценным координатором набора задач.

Новые операции:

- `listTasks()`
- `activeTask()`
- `switchTask(taskId)`
- `resumeTask(taskId)`
- `pauseActiveTask()`
- `completeTask(taskId)`

Важные инварианты:

- активной может быть только одна задача;
- `DONE` задача не может быть активной;
- `PAUSED` задача не считается активной;
- при переключении старая активная задача приводится к согласованному статусу.

Дополнительные возможные абстракции:

- `ActiveTaskResolver`
- `TaskSummaryView`
- `TaskSelectionPolicy`
- возможное расширение `TaskStageDefinition`, если появятся новые стартовые стадии или дополнительные display-поля

Prompt/orchestration:

- в task context идёт только активная задача;
- paused/done задачи не подмешиваются как текущий активный workflow.

### Вероятные места интеграции

- task core/persistence/model
- CLI parser/controller/renderer/help
- agent orchestration

### Как понять, что этап готов

Функционально:

- можно создать несколько задач;
- можно переключаться между ними;
- активной всегда остаётся только одна;
- у каждой задачи сохраняются свои `stage`, `step`, `expected action`;
- агент ориентируется именно на активную задачу.

Что проверить вручную:

1. Создать 2-3 задачи.
2. Переключаться между ними.
3. Проверить, что активная всегда одна.
4. Возобновить paused-задачу.
5. Завершить одну задачу.
6. Проверить, что остальные продолжают жить корректно.
7. Переключить active user.
8. Проверить, что task list и active task не меняются автоматически.

Какие тесты нужны:

- switch корректно меняет активную задачу;
- resume переводит paused-задачу в active;
- done убирает задачу из активных;
- инвариант “одна активная” не нарушается.

## Этап 5. UX и повседневная удобность

### Смысл этапа

Сделать task system удобной для реальной работы, а не только формально правильной.

### Что видит пользователь

- удобный `/task show`;
- удобный `/task list`;
- более понятные статусы;
- возможно пошаговый `/task start`;
- возможно короткие task-подсказки после ответов.

### Что пользователь может делать

- быстро понимать, на чём остановились;
- быстро видеть `expected action`;
- меньше помнить команды и внутреннюю структуру.

Примеры:

- “На чём мы остановились?” → быстрый понятный вывод по активной задаче.
- После `resume` сразу видно:
  - stage
  - current step
  - expected action

### Что это даёт пользователю

- task system становится естественной в ежедневном использовании;
- снижается когнитивная нагрузка;
- возобновление старых задач становится быстрым и понятным.

### Технические изменения

Это в основном presentation/UI-слой.

Возможные абстракции:

- `TaskCliCatalog`
- `TaskViewModel`
- расширения для текущего renderer
- если будет мастер, то `TaskDraft`

Если делаем пошаговое создание:

- draft-state логика в CLI по аналогии с memory note draft

Важно:

- доменная модель здесь почти не должна ломаться;
- меняем в основном UX поверх уже стабильного core.

### Как понять, что этап готов

Функционально:

- команды дают компактный и понятный вывод;
- ключевые task workflows проходятся почти без help;
- resume и show действительно помогают быстро восстановить контекст.

Что проверить вручную:

1. Пройти основной сценарий, почти не подглядывая в help.
2. Возобновить старую задачу и понять контекст за 1-2 команды.
3. Проверить читаемость всех task-related сообщений.

Какие тесты нужны:

- parser/controller на новые UX-команды;
- проверки форматирования вывода.

## Этап 6. Умные предложения от агента, но без автопереключений

### Смысл этапа

Добавить умные предложения, не ломая главный принцип: task state не меняется без подтверждения пользователя.

### Что видит пользователь

Агент начинает замечать типичные переходы и предлагать действия:

- “Похоже, началась новая задача”
- “Похоже, мы перешли к validation”
- “Похоже, сейчас ожидается решение пользователя”
- “Хочешь поставить текущую задачу на паузу?”

Но:

- агент не меняет task state молча;
- агент не создаёт и не переключает задачи автоматически.

### Что пользователь может делать

- принимать предложение;
- отклонять предложение;
- по-прежнему явно контролировать task state.

### Что это даёт пользователю

- меньше ручной рутины;
- система становится помощником, но не самовольным диспетчером.

### Технические изменения

Нужен отдельный механизм предложений, не связанный с `pending`.

Новые абстракции:

- `TaskSuggestion`
- `TaskSuggestionType`
- `TaskSuggestionService`

Примеры suggestion types:

- `START_NEW_TASK`
- `PAUSE_CURRENT_TASK`
- `RESUME_TASK`
- `UPDATE_EXPECTED_ACTION`
- `TRANSITION_STAGE`

Что делает сервис:

- анализирует сообщение пользователя и текущий task state;
- формирует предложение;
- UI показывает его пользователю;
- применение предложения происходит только после подтверждения.

Важно:

- это не memory allocator;
- это не `pending`;
- это отдельный flow task orchestration.

### Как понять, что этап готов

Функционально:

- предложения появляются в уместных ситуациях;
- без подтверждения task state не меняется;
- после подтверждения изменение происходит корректно.

Что проверить вручную:

1. Спровоцировать ситуацию, похожую на новый рабочий трек.
2. Проверить, что агент предлагает, а не переключает сам.
3. Отказаться и убедиться, что состояние не изменилось.
4. Подтвердить и проверить, что состояние обновилось правильно.

Какие тесты нужны:

- suggestions формируются в уместных сценариях;
- без подтверждения состояние не меняется;
- применение suggestion корректно обновляет task state.

## Рекомендуемый порядок внедрения

Лучший порядок остаётся таким:

1. Этап 1
2. Этап 2
3. Этап 3
4. Этап 4
5. Этап 5
6. Этап 6

Почему:

- после этапа 2 уже получится полезная первая версия;
- этапы 3-4 доводят систему до целевой multitask-модели;
- этапы 5-6 улучшают удобство и интеллектуальность.

## Критерии готовности этапа в общем виде

Для каждого этапа полезно проверять 3 уровня:

1. `Domain`
   - модели и инварианты реализованы корректно

2. `Persistence`
   - состояние сохраняется и восстанавливается правильно

3. `UX / Behavior`
   - пользователь видит ожидаемое поведение
   - агент учитывает task state там, где это уже предусмотрено этапом

## Короткое резюме ценности по этапам

- Этап 1: появляется одна формализованная задача
- Этап 2: агент начинает ей подчиняться
- Этап 3: задачи становятся отдельными сохраняемыми треками
- Этап 4: появляется полноценная модель “несколько задач, одна активная”
- Этап 5: пользоваться этим становится удобно
- Этап 6: агент начинает умно помогать, но не теряет управляемость
