# Task State Stage 4 Breakdown

## Цель этапа

Довести task subsystem до полноценной пользовательской модели:

- в системе может быть несколько задач;
- в каждый момент времени активна только одна;
- пользователь умеет явно смотреть список задач, переключаться между ними, возобновлять paused-задачи и завершать конкретную задачу по `id`;
- агент и prompt/orchestration по-прежнему работают только с активной задачей.

На выходе этапа:

- multitask-модель становится видимой не только внутри домена, но и в CLI;
- у пользователя появляется явный контроль над активной задачей;
- инвариант «одна активная задача» соблюдается и в домене, и в UX, и после restart.

## Что входит в этап 4

- полноценные multitask-команды в CLI;
- расширение `TaskManager` до явных операций над конкретными задачами;
- явные правила переключения active task;
- отображение списка задач и активной задачи;
- сохранение и восстановление multitask session state;
- тесты и smoke-проверка полного multitask-flow.

## Что не входит в этап 4

- умные предложения от агента по переключению задач;
- автоматическое создание или переключение задач по обычному сообщению;
- user-scoped tasks;
- богатый UX поверх multitask-модели;
- сложные политики выбора задач beyond one-active-task invariant.

## Блок 1. Domain / Invariants

### Подзадача 1.1. Зафиксировать stage 4 инварианты multitask-модели

**ТЗ**

Нужно явно оформить целевой контракт этапа 4:

- в session может быть 0..N задач;
- активной может быть только одна задача;
- `activeTaskId` либо указывает на существующую задачу, либо равен `null`;
- `done` задача не может считаться активной;
- `paused` задача не активна, но доступна для возобновления;
- переключение active task не должно уничтожать другие задачи;
- агент и prompt используют только активную задачу.

Нужно также определить ожидаемое поведение для операций:

- `switchTask(taskId)`
- `resumeTask(taskId)`
- `pauseActiveTask()`
- `completeTask(taskId)`

**Результат**

Есть понятный доменный контракт этапа 4, на который будут опираться manager, CLI и тесты.

### Подзадача 1.2. Уточнить согласованное состояние задач при переключении

**ТЗ**

Нужно зафиксировать, что происходит с предыдущей активной задачей при переключении:

- при `switchTask(taskId)` прежняя активная задача, если она не `done`, переводится в `paused`;
- выбранная задача становится `active`;
- если выбранная задача была `paused`, она корректно переводится в `active`;
- если выбранная задача уже `done`, операция либо запрещается, либо даёт понятный отказ;
- если переключение идёт на уже активную задачу, операция должна вести себя предсказуемо и не ломать state.

Важно:

- не вводить скрытых побочных эффектов;
- не создавать вторую активную задачу ни в одном сценарии;
- не завязываться на UI, а зафиксировать именно доменные правила.

**Результат**

Поведение task subsystem при переключении задач становится формализованным и однозначным.

## Блок 2. Task Manager API

### Подзадача 2.1. Расширить `TaskManager` операциями stage 4

**ТЗ**

Нужно добавить в `TaskManager` и `DefaultTaskManager` явные multitask-операции:

- `listTasks()`
- `activeTask()`
- `switchTask(taskId)`
- `resumeTask(taskId)`
- `completeTask(taskId)`

Если текущие single-active методы уже есть, их нужно сохранить только там, где это упрощает переход, но основной stage 4 API должен быть явным и ориентированным на `taskId`.

Важно:

- не ломать текущую работу агента с активной задачей;
- не дублировать логику переключения между CLI и manager;
- весь state transition делать в одном месте, внутри task core.

**Результат**

Task subsystem получает полноценный API для управления несколькими задачами.

### Подзадача 2.2. Реализовать строгий инвариант «одна активная задача»

**ТЗ**

Нужно убедиться, что manager не допускает неконсистентных состояний:

- после `startTask(...)` активной остаётся только новая задача;
- после `switchTask(...)` активной остаётся только выбранная задача;
- после `resumeTask(...)` активной становится только возобновлённая задача;
- после `completeTask(activeTaskId)` активная задача либо снимается, либо определяется по согласованному правилу;
- состояние без активной задачи тоже обрабатывается корректно.

Важно:

- не размазывать эту логику по CLI и persistence;
- manager должен быть источником истины по инварианту active task.

**Результат**

Даже при разных пользовательских сценариях внутри домена всегда сохраняется только одна активная задача.

## Блок 3. CLI Surface

### Подзадача 3.1. Добавить multitask-команды

**ТЗ**

Нужно расширить CLI-команды до stage 4 уровня.

Минимальный набор:

- `/task list`
- `/task switch <id>`
- `/task resume <id>`
- `/task done <id>`

При необходимости допустимо сохранить:

- `/task show` как показ активной задачи;
- `/task done` без `id` как shorthand для active task, только если это не создаёт путаницу.

Важно:

- команды должны быть явными и читаемыми;
- ошибки должны быть понятными: неизвестный `id`, попытка возобновить `done`, отсутствие active task и т.д.;
- parser и controller не должны повторять бизнес-логику manager.

**Результат**

Пользователь получает минимально достаточный multitask UX для работы с несколькими задачами.

### Подзадача 3.2. Сделать понятное отображение списка задач

**ТЗ**

Нужно добавить user-facing представление для `/task list`.

В выводе желательно показывать:

- `id`
- `title`
- `status`
- `stage`
- маркер активной задачи

Важно:

- вывод должен быть компактным;
- активная задача должна читаться сразу;
- paused и done задачи не должны теряться;
- не превращать stage 4 в stage 5 с избыточным polish.

**Результат**

`/task list` становится рабочим инструментом, а не сырой dump структуры.

## Блок 4. Agent / Prompt Compatibility

### Подзадача 4.1. Сохранить active-task-only contract для prompt и orchestration

**ТЗ**

После появления multitask UX нужно проверить и при необходимости доработать интеграции так, чтобы:

- `TaskPromptContext` строился только по активной задаче;
- `TaskOrchestrationService` принимал только активную задачу;
- paused и done задачи не подмешивались в `system message`;
- список задач не попадал в prompt целиком.

Важно:

- stage 4 не должен ломать уже реализованное task-aware поведение;
- multitask-модель должна быть видна пользователю, но не должна засорять prompt для модели.

**Результат**

Расширение task subsystem до нескольких задач не ломает качество task-aware prompt/orchestration.

### Подзадача 4.2. Проверить согласованность поведения после `switch` и `resume`

**ТЗ**

Нужно убедиться, что после пользовательских действий:

- `switch`
- `resume`
- `done <id>`

агент ориентируется именно на новую активную задачу, а не на ранее активную.

Нужно проверить:

- `show` и `list` показывают одно и то же active state;
- prompt после переключения отражает именно новую active task;
- `paused` старые задачи не влияют на поведение агента.

**Результат**

Есть уверенность, что UX stage 4 и task-aware behavior согласованы между собой.

## Блок 5. Verification

### Подзадача 5.1. Покрыть stage 4 тестами

**ТЗ**

Нужны тесты минимум на такие сценарии:

- `listTasks()` возвращает корректный набор задач;
- `switchTask(taskId)` корректно меняет active task;
- `resumeTask(taskId)` переводит paused-задачу в active;
- `completeTask(taskId)` корректно меняет статус и active state;
- инвариант «одна активная задача» не нарушается;
- persistence корректно хранит и восстанавливает несколько задач с `activeTaskId`;
- prompt/orchestration используют только активную задачу после `switch` и `resume`.

**Результат**

Контракт stage 4 зафиксирован тестами.

### Подзадача 5.2. Подготовить smoke-сценарий stage 4

**ТЗ**

Нужен scripted smoke-check, который проходит полный user-facing multitask flow.

Базовый сценарий:

1. `/task start Первая задача`
2. `/task pause`
3. `/task start Вторая задача`
4. `/task list`
5. `/task switch task-1`
6. `/task show`
7. `/task resume task-2`
8. `/task done task-1`
9. `/task list`
10. перезапуск приложения
11. `/task list`
12. `/task show`

При необходимости сценарий можно уточнить под фактические `id` и итоговый UX.

Важно:

- setup и verify запускать строго последовательно;
- ориентироваться на `DebugTraceListener` и trace/output, а не на сырой CLI stdout.

**Результат**

Stage 4 подтверждён не только тестами, но и реальным multitask restart-flow.

## Критерии готовности этапа 4

- пользователь может создать несколько задач и увидеть их списком;
- пользователь может переключаться между задачами и возобновлять paused-задачи по `id`;
- активной всегда остаётся только одна задача;
- done-задачи не становятся активными случайно;
- prompt и orchestration используют только активную задачу;
- multitask state корректно переживает restart;
- stage 4 подтверждён тестами и smoke-сценарием.
