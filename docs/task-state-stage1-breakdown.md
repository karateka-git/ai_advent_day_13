# Task State Stage 1 Breakdown

Этот документ детализирует первый этап из [task-state-roadmap.md](./task-state-roadmap.md) в виде крупных блоков работ.

Граница этапа 1:

- одна текущая задача;
- task state меняется только через `/task`-команды;
- несколько задач не поддерживаются;
- жёсткой валидации переходов между stage нет;
- task list считается `conversation-scoped`;
- переключение `active user` не меняет task state автоматически.

## Блок 1. Domain

### Цель блока

Ввести минимальную доменную модель task subsystem и API управления одной текущей задачей.

### Подзадача 1.1. Зафиксировать контракт этапа 1

#### ТЗ

Нужно зафиксировать минимальный контракт первой реализации task state:

- одна текущая задача;
- поля: `title`, `stage`, `currentStep`, `expectedAction`, `status`;
- richer-стартовый набор stage: `planning`, `execution`, `validation`, `completion`;
- завершение задачи хранится в `status = done`;
- task state меняется только через `/task`;
- обычные сообщения пользователя не меняют task state;
- несколько задач, suggestions и строгая transition validation не входят в scope этапа 1.

#### Результат

Есть единый implementation scope, на который опираются код, тесты и ручная проверка.

#### Готово, когда

- нет противоречий между roadmap и stage 1 breakdown;
- доменная модель не предполагает multitask;
- явно зафиксировано, что `done` — это статус, а не stage.

### Подзадача 1.2. Ввести доменные типы task subsystem

#### ТЗ

Нужно создать минимальные типы в `agent.task.model`:

- `TaskStage`
- `TaskStageDefinition`
- `TaskStages`
- `TaskStatus`
- `ExpectedAction`
- `TaskState`

Требования:

- `TaskStage` — enum со стартовыми значениями `PLANNING`, `EXECUTION`, `VALIDATION`, `COMPLETION`;
- `TaskStatus` — enum со значениями `ACTIVE`, `PAUSED`, `DONE`;
- `ExpectedAction` — enum со значениями `USER_INPUT`, `AGENT_EXECUTION`, `USER_CONFIRMATION`, `NONE`;
- `TaskState` хранит все поля текущей задачи в типизированном виде;
- `ExpectedAction` не дублирует `TaskStage`.

#### Результат

В проекте появляется самостоятельная typed-модель task subsystem.

#### Готово, когда

- все перечисленные типы созданы;
- значения enum отражают текущие договорённости roadmap;
- в модели нет смешения `done` как stage и `done` как status.

### Подзадача 1.3. Реализовать B-lite каталог стадий

#### ТЗ

Нужно реализовать `TaskStageDefinition` и `TaskStages` как единый каталог metadata для stage.

Минимально каталог должен уметь:

- давать label для stage;
- давать короткое описание stage;
- быть единым источником семантики stage для CLI, prompt и orchestration.

Требования:

- metadata не должна размазываться по `when`-веткам по всему проекту;
- при добавлении stage в будущем изменения должны концентрироваться в одном месте.

#### Результат

Появляется `B-lite`-слой, который позволяет стартовать с enum и не запирать развитие модели.

#### Готово, когда

- `TaskStages` покрывает все стартовые stage;
- CLI и task-related presentation могут брать label/description из definitions;
- каталог стадий можно использовать как единый источник истины.

### Подзадача 1.4. Реализовать core API для одной задачи

#### ТЗ

Нужно реализовать сервис управления одной текущей задачей, например:

- `TaskManager`
- или `DefaultTaskManager`

Минимальный набор операций:

- `currentTask()`
- `startTask(title: String)`
- `updateStage(stage: TaskStage)`
- `updateStep(step: String)`
- `updateExpectedAction(action: ExpectedAction)`
- `pauseTask()`
- `resumeTask()`
- `completeTask()`
- `clearTask()`

Требования:

- строгая валидация переходов между stage пока не нужна;
- должны быть базовые sanity-check;
- при отсутствии текущей задачи изменения task state должны завершаться понятной ошибкой;
- `completeTask()` должен переводить задачу в `status = DONE`.

#### Результат

Есть минимальный core-сервис, которым смогут пользоваться CLI, persistence и агент.

#### Готово, когда

- можно создать, прочитать, обновить, поставить на паузу, снять с паузы, завершить и очистить текущую задачу;
- API не завязано на CLI или UI;
- поведение `completeTask()` и `pauseTask()/resumeTask()` однозначно определено.

## Блок 2. Persistence

### Цель блока

Сделать task state устойчивым между перезапусками и привязанным к текущей conversation session.

### Подзадача 2.1. Спроектировать storage model task state

#### ТЗ

Нужно определить, как task state живёт в persistence на этапе 1.

Требования:

- task state считается `conversation-scoped`;
- он должен сохраняться и восстанавливаться вместе с состоянием текущей сессии;
- переключение `active user` не должно автоматически менять task state;
- storage model должна быть простой и совместимой с single-task этапом.

#### Результат

Есть ясная storage policy для первой версии task subsystem.

#### Готово, когда

- зафиксировано, где именно хранится task state;
- понятно, как он загружается при старте;
- понятно, что происходит при отсутствии сохранённого task state.

### Подзадача 2.2. Реализовать repository

#### ТЗ

Нужно реализовать persistence-абстракцию:

- `TaskStateRepository`
- `JsonTaskStateRepository`

Repository должен уметь:

- загружать текущее состояние задачи;
- сохранять текущее состояние задачи;
- очищать состояние задачи, если нужно.

Требования:

- storage не должен притворяться memory subsystem;
- task persistence должна жить отдельно от `agent.memory.*`, даже если физически использует близкие project paths;
- структура JSON должна быть понятной и пригодной для отладки.

#### Результат

Task state переживает перезапуск приложения.

#### Готово, когда

- `startTask()` + restart + `currentTask()` восстанавливают состояние;
- `pause/resume/done` тоже корректно сохраняются;
- очистка task state не оставляет ложных артефактов.

### Подзадача 2.3. Подключить persistence к core

#### ТЗ

Нужно связать `TaskManager` и `TaskStateRepository`.

Требования:

- при старте manager должен загружать persisted task state;
- после изменения task state manager должен сохранять его;
- отсутствие сохранённого состояния должно корректно трактоваться как “текущей задачи нет”.

#### Результат

Core и persistence работают как единая task subsystem.

#### Готово, когда

- все mutating-операции task manager сохраняют результат;
- повторный запуск приложения поднимает последнее состояние;
- при пустом storage система работает корректно.

## Блок 3. CLI

### Цель блока

Дать пользователю явный интерфейс управления одной текущей задачей.

### Подзадача 3.1. Добавить parser и command model для `/task`

#### ТЗ

Нужно расширить CLI-модель и parser новым набором команд task subsystem.

Минимальный набор:

- `/task start <title>`
- `/task show`
- `/task stage <value>`
- `/task step <text>`
- `/task expect <value>`
- `/task pause`
- `/task resume`
- `/task done`
- `/task clear`

Дополнительно можно поддержать:

- `/task help`

Требования:

- команды должны быть однозначными;
- значения `stage` и `expected action` должны маппиться в typed enum;
- парсер не должен менять task state сам по себе, только описывать команду.

#### Результат

CLI умеет распознавать task-команды первого этапа.

#### Готово, когда

- все команды распознаются парсером;
- ошибки формата обрабатываются читаемо;
- обычный текст по-прежнему не трактуется как task mutation.

### Подзадача 3.2. Подключить task-команды к controller

#### ТЗ

Нужно научить `CliSessionController` выполнять `/task`-команды через `TaskManager`.

Требования:

- все изменения task state идут только через manager;
- ошибки отсутствующей задачи, неправильного ввода и пустых значений показываются через обычный CLI error flow;
- обычные сообщения пользователя без `/task` не меняют task state.

#### Результат

Пользователь может управлять одной задачей полностью через CLI.

#### Готово, когда

- каждая `/task`-команда вызывает соответствующее действие core-сервиса;
- ошибки и success-сообщения выглядят согласованно;
- state не меняется в обход controller.

### Подзадача 3.3. Сделать человекочитаемый вывод task state

#### ТЗ

Нужно реализовать отображение `/task show`.

Показывать минимум:

- title
- stage
- current step
- expected action
- status

Требования:

- использовать `TaskStages` для label/description;
- вывод должен быть компактным и читаемым;
- пользователь должен понимать состояние задачи за один экран без чтения JSON.

#### Результат

`/task show` становится полезным пользовательским инструментом, а не просто dump структуры.

#### Готово, когда

- отображаются все ключевые поля;
- stage показывается через нормальный label;
- paused/done состояние видно явно.

## Блок 4. Context

### Цель блока

Подмешать task state в агентский контекст без жёсткой оркестрации этапа 2.

### Подзадача 4.1. Реализовать task prompt assembler

#### ТЗ

Нужно сделать task-derived prompt context, который `TaskManager` сможет отдавать для модели.

Минимально в блок должны входить:

- title
- stage
- current step
- expected action
- status

Требования:

- если текущей задачи нет, task block не подмешивается;
- если задача есть, её описание должно быть кратким и стабильным;
- `expected action` должно интерпретироваться как ближайшее нужное действие, а не как вторая версия stage.

#### Результат

Task state становится видимым для модели уже на этапе 1.

#### Готово, когда

- assembler корректно формирует task block;
- отсутствие задачи не ломает prompt;
- task context можно использовать отдельно от memory prompt assembly.

### Подзадача 4.2. Подключить task context к агенту

#### ТЗ

Нужно интегрировать task context в сборку итогового prompt или соседний orchestration-слой.

Требования:

- task subsystem не должна смешиваться с memory subsystem;
- wiring должен идти через bootstrap/factory;
- на этапе 1 достаточно “видимости task context” для модели, без полного enforce поведения этапа 2.

#### Результат

Агент получает контекст текущей задачи ещё до полноценного task-aware поведения.

#### Готово, когда

- task context реально попадает в conversation/prompt;
- это не ломает существующую memory assembly;
- обычный runtime без текущей задачи продолжает работать как раньше.

## Блок 5. Verification

### Цель блока

Зафиксировать контракт этапа 1 тестами и ручным сценарием.

### Подзадача 5.1. Написать unit и integration tests

#### ТЗ

Нужно покрыть тестами:

- создание задачи;
- обновление stage;
- обновление step;
- обновление expected action;
- pause/resume;
- done;
- clear;
- save/load;
- parser `/task`-команд;
- controller task flow;
- базовый task prompt assembly.

Требования:

- тесты должны фиксировать именно контракт этапа 1;
- multitask-сценарии не должны утекать в этот слой раньше времени;
- тесты не должны завязываться на будущие suggestions или строгую transition validation.

#### Результат

Первая версия task subsystem защищена от регрессий.

#### Готово, когда

- ключевые task use-cases покрыты тестами;
- run тестов подтверждает контракт этапа 1;
- тесты читаемо отражают, что именно считается допустимым поведением.

### Подзадача 5.2. Пройти ручной сценарий этапа 1

#### ТЗ

Нужно прогнать ручной сценарий:

1. `/task start Реализовать task subsystem`
2. `/task stage planning`
3. `/task step Продумать доменную модель`
4. `/task expect user_input`
5. `/task show`
6. перезапуск приложения
7. `/task show`
8. `/task pause`
9. обычное сообщение без `/task`
10. убедиться, что paused-задача не возобновилась автоматически
11. `/task resume`
12. `/task stage completion`
13. `/task show`
14. `/task done`
15. `/task show`
16. переключить `active user`
17. проверить, что task state не поменялся автоматически

Требования:

- сценарий должен подтверждать conversation-scoped модель;
- должен проверять `completion` как stage и `done` как status;
- должен проверять, что обычный текст не меняет task state.

#### Результат

Есть реальное подтверждение, что stage 1 работает с пользовательской точки зрения.

#### Готово, когда

- весь сценарий проходит без ручных обходов;
- состояние переживает restart;
- `active user` не ломает task state;
- вывод CLI понятен пользователю.

## Что не делаем в этапе 1

- несколько задач;
- `activeTaskId` и multitask UX;
- suggestions от агента;
- user-scoped tasks;
- жёсткую валидацию переходов между stage;
- автоматическое изменение task state из обычных пользовательских сообщений.
