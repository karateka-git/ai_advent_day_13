# Task State Stage 3 Breakdown

## Цель этапа

Подготовить внутреннюю модель к нескольким задачам, даже если пользовательский UX пока ещё не стал полноценным multitask.

На выходе этапа:

- задачи хранятся списком, а не одной записью;
- есть явный `activeTaskId`;
- создание новой задачи не уничтожает старую;
- persisted state умеет хранить и восстанавливать несколько задач;
- агент по-прежнему работает только с одной активной задачей.

## Что входит в этап 3

- переход от single-task storage к task session state;
- новый доменный контейнер для списка задач;
- обновление persistence под список задач;
- адаптация `TaskManager` к работе с набором задач;
- сохранение текущего UX по возможности минимальным;
- тесты и smoke-проверка внутренней модели нескольких задач.

## Что не входит в этап 3

- полноценные команды `/task list`, `/task switch`, `/task resume <id>`;
- богатый multitask UX;
- автопредложения от агента;
- user-scoped tasks;
- сложные политики выбора активной задачи.

## Блок 1. Domain Model

### Подзадача 1.1. Ввести session-модель задач

**ТЗ**

Нужно заменить внутреннюю модель “одна задача или null” на модель с контейнером задач.

Новые сущности:

- `TaskItem`
- `TaskSessionState`

Минимальный состав:

- `TaskItem`
  - `id`
  - `title`
  - `stage`
  - `status`
  - `currentStep`
  - `expectedAction`
- `TaskSessionState`
  - `tasks`
  - `activeTaskId`

Важно:

- `TaskItem` на этом этапе может быть близок к текущему `TaskState`;
- `TaskState` можно либо убрать, либо превратить в alias/переходную модель, если это не усложняет код;
- активная задача должна находиться явно, а не “по косвенным признакам”.

**Результат**

В домене появляется явная модель task session state, рассчитанная на несколько задач.

### Подзадача 1.2. Зафиксировать инварианты stage 3

**ТЗ**

Нужно явно оформить базовые инварианты новой модели:

- в session может быть 0..N задач;
- активная задача может быть только одна;
- `activeTaskId` либо указывает на существующую задачу, либо равен `null`;
- `done` задача не должна становиться активной случайно;
- paused и done задачи не теряются при создании новой задачи.

На этапе 3 эти инварианты могут пока обеспечиваться через manager/service-слой без отдельной жёсткой state machine.

**Результат**

Есть понятный набор правил, на которые потом опираются persistence и `TaskManager`.

## Блок 2. Persistence

### Подзадача 2.1. Перейти от single-task repository к session repository

**ТЗ**

Нужно заменить persistence одной задачи на persistence task session state.

Ожидаемые изменения:

- новый контракт `TaskSessionStateRepository`;
- JSON-реализация для списка задач и `activeTaskId`;
- корректный save/load для пустой session;
- корректный save/load для нескольких задач.

Важно:

- без legacy migration и backward compatibility;
- считаем, что проект в активной разработке и можно хранить только новый формат;
- storage остаётся conversation-scoped.

**Результат**

Task subsystem сохраняет не одну задачу, а весь task session state.

### Подзадача 2.2. Убедиться, что restart не ломает активную задачу

**ТЗ**

После перехода на session persistence нужно проверить:

- новая задача сохраняется;
- paused старая задача тоже сохраняется;
- активная задача корректно восстанавливается после restart;
- отсутствие активной задачи тоже восстанавливается корректно.

**Результат**

Persistence stage 3 пригодна для реального restart-flow.

## Блок 3. Task Manager

### Подзадача 3.1. Перевести `TaskManager` на session state

**ТЗ**

Нужно обновить `TaskManager` и `DefaultTaskManager`, чтобы они работали не с одним `TaskState`, а с `TaskSessionState`.

Минимальные операции этапа 3:

- `currentTask()`
- `startTask(title)`
- `updateStage(...)`
- `updateStep(...)`
- `updateExpectedAction(...)`
- `pauseTask()`
- `resumeTask()`
- `completeTask()`
- `clearTask()`

Поведение `startTask(title)` на этапе 3:

- если активной задачи нет, новая становится активной;
- если активная задача есть, она не удаляется;
- новая задача становится активной;
- прежняя активная приводится к согласованному состоянию.

Допустимое временное решение:

- старая активная задача автоматически переводится в `paused`, если ещё не `done`.

**Результат**

`TaskManager` уже внутренне multitask-ready, даже если CLI пока этого почти не показывает.

### Подзадача 3.2. Минимально сохранить совместимость текущего Agent API

**ТЗ**

Поскольку UX этапа 3 ещё не про полный multitask, важно не ломать наружный контракт без необходимости.

На этом этапе допустимо:

- сохранить методы, работающие с “текущей задачей”;
- внутри менеджера считать их операциями над активной задачей;
- не тащить в `Agent` список задач раньше времени, если это не нужно для минимального результата.

**Результат**

Внутренний переход на session state не требует преждевременного расширения пользовательского API.

## Блок 4. Minimal UX Surface

### Подзадача 4.1. Уточнить поведение `/task start`

**ТЗ**

Нужно сделать поведение создания новой задачи понятным даже без команд list/switch.

Минимум для этапа 3:

- `/task start ...` создаёт новую активную задачу;
- предыдущая активная задача не исчезает;
- если предыдущая задача была активной и не завершённой, она переводится в `paused`;
- `/task show` продолжает показывать только текущую активную задачу.

Важно:

- не делать ещё полноценный multitask UI;
- не пытаться показать все задачи раньше этапа 4;
- но не терять сохранённые задачи.

**Результат**

Пользователь уже чувствует, что новая задача не стирает старую, даже при минимальном UX.

### Подзадача 4.2. Не сломать orchestration этапа 2

**ТЗ**

После перехода к нескольким задачам агент всё ещё должен опираться только на активную задачу.

Нужно проверить:

- `TaskOrchestrationService` получает активную задачу;
- prompt contribution строится только из активной задачи;
- paused и done неактивные задачи не смешиваются в текущий task context.

**Результат**

Модель нескольких задач не ломает уже реализованное task-aware поведение агента.

## Блок 5. Verification

### Подзадача 5.1. Покрыть stage 3 тестами

**ТЗ**

Нужны тесты минимум на такие сценарии:

- создание второй задачи не удаляет первую;
- старая активная задача после `startTask(...)` новой переводится в `paused`;
- `activeTaskId` меняется корректно;
- save/load нескольких задач работает;
- agent/task prompt продолжает использовать только активную задачу.

**Результат**

Контракт stage 3 зафиксирован тестами.

### Подзадача 5.2. Подготовить smoke-сценарий stage 3

**ТЗ**

Нужен smoke-сценарий, который подтверждает именно внутреннюю multitask-модель, даже если UX ещё минимальный.

Базовый сценарий:

1. `/task start Первая задача`
2. `/task pause`
3. `/task start Вторая задача`
4. `/task show`
5. Перезапуск приложения
6. `/task show`
7. Проверка persisted task storage

Если к моменту реализации будет доступен минимальный debug-вывод task session state, можно использовать его в smoke-check.

**Результат**

Stage 3 подтверждён не только тестами, но и реальным restart-flow.

## Критерии готовности этапа 3

- внутри системы хранится список задач и `activeTaskId`;
- создание новой задачи не удаляет предыдущую;
- активная задача восстанавливается после restart;
- агент ориентируется только на активную задачу;
- stage 3 подтверждён тестами и smoke-сценарием.
